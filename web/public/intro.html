IndiStream. № 09 / 13 · algorithm · 1 of 4

Algorithm step 1

Embed and retrieve.

The user's brief becomes a 1536-dimensional vector via OpenAI's text-embedding-3-small. The same model embedded every work's title + description at seed time, so query and corpus live in the same space.

Retrieval runs in Postgres via pgvector. We rank candidates by cosine similarity against the query vector and return the top 50, then hydrate each row with its artist’s tier and the work’s price. The next stages (filter, score, rerank) all run in-process on this hydrated set — one DB round-trip per search.

Index = HNSW, not IVFFlat. We hit a real bug switching: with IVFFlat (lists=100) on a ~150-row corpus, default probe count pulled zero rows. HNSW has no equivalent failure on small data. The migration is one statement.

SQL · the RPC 
create or replace function
  public.search_works(
    query_embedding vector(1536),
    match_count int default 50
  )
returns table (
  work_id uuid, artist_id uuid,
  title text, medium medium_kind,
  similarity float
)
language sql stable
as $$
  select
    w.id, w.artist_id, w.title, w.medium,
    1 - (w.embedding <=> query_embedding)
      as similarity
  from public.works w
  where w.embedding is not null
  order by w.embedding <=> query_embedding
  limit match_count;
$$;
IndiStream. № 10 / 13 · algorithm · 2 of 4

Algorithm step 2

Score, don’t just rank.

Hard filters first. Medium mismatch, missing verification, or price beyond 1.5× the buyer’s stated budget drops a candidate before scoring. Cheap, predictable, lets the buyer trust the chips they ticked.

Among survivors, every work gets a weighted score:

score = α·rel + δ·health − ε·price_pen

α (1.0) — pure cosine relevance.
δ (0.5) — tier health bonus (emerging 1.0 / growing 0.5 / established 0.0). Gated by rel ≥ 0.20 so we don’t surface low-relevance “emerging” results just for the bonus.
ε (0.6) — squared price penalty: 0 under budget, ramps to 1 at 1.5×, where the hard filter already rejected it.
β = γ = 0 in v1 — revenue and embedding-MMR diversity are weights reserved for v2.

β  beta_revenue     = 0.0   ← v2
γ  gamma_diversity  = 0.0   ← v2

IndiStream. № 11 / 13 · algorithm · 3 of 4

Algorithm step 3

Spread, then guarantee.

Round-robin by artist (kept from v0). Pull one work per artist until each bucket drains, preserving each bucket’s score order. Stops one creator from monopolising the top-K when their style genuinely matches the brief.

Quota repair pass. After round-robin, count how many top-K rows belong to emerging artists with rel ≥ 0.40. If fewer than ⌈K × 0.2⌉, swap qualifying emerging candidates in. Repaired rows ship with quota_repaired: true so the UI can mark them.

Why the second pass? Round-robin alone biases toward established artists — they have more works in the candidate pool, so they win more buckets. Quota repair forces long-tail visibility without letting low-relevance results sneak through (the 0.40 floor is hard).

Sister logic in artist-mode: artist_score = mean(top-3 work similarities). Same scoring/filtering policy applied per-artist; no separate bio embeddings yet — that’s the obvious next addition once onboarding lets artists sign up before listing.

IndiStream. № 12 / 13 · algorithm · 4 of 4

Algorithm step 4 — signaling & UX

Honest results, deferred filters.

Confidence tags. Every returned row is bucketed by its raw relevance: HIGH ≥ 0.55, MEDIUM ≥ 0.40, LOW ≥ 0.25, else NO_MATCH. The UI uses this to dim, badge, or refuse to show — so a brief with no real match stays honest instead of forcing a top-10.

Deferred apply. Prompt edits, mode toggle, budget chips, and verified all stage; nothing fetches until Apply. Apply commits the whole batch in one round-trip and fires the thinking transition.

Two channels for refinement. Structured chips (medium, budget, verified) become hard filter parameters — exact, predictable. Free-form feel words (“warmer”, “more tape”) get appended to the query text and embedded — soft, semantic. Buyers rarely realise both layers exist; they just feel composable.
